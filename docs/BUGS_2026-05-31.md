# Open Bugs — 2026-05-31

Three reproducible UI bugs in the current `tui` / `renderer` code. Each section
below is self-contained: symptom, root cause (with `file:line` references), a
suggested fix, and the verification / golden-file steps a future agent should
follow. The bugs are independent and can be fixed in any order, though Bug 3 is
the largest and touches the most files.

| # | Area | One-line summary |
|---|------|------------------|
| 1 | `renderer/pile.go` | Start-menu background shows tableau columns as 1-line stubs instead of a full face-down top card |
| 2 | `tui/cursor.go` | Foundation piles are unreachable with the arrow keys |
| 3 | `tui/board.go` + `renderer` | Keyboard `Enter` selection teleports the card to the top-left (0,0) and renders no source/destination indicator |

---

## Bug 1 — Menu background tableau renders as stubs, not full face-down cards

### Symptom
When the start menu is open, the decorative board drawn behind it shows each
tableau column as a stack of thin one-row stubs (`▇▇▇▇▇▇▇`). No actual card body
is visible. The expectation is that the **top** (bottom-most, most-accessible)
card of every tableau pile renders as a *full* face-down card (the 5-row
`▇`/`█` block), with the cards beneath it shown as stubs — exactly how a real
Klondike layout reads.

### How to reproduce
1. `go run .`
2. Observe the board behind the centered settings/start menu box. The tableau
   columns peeking out from the left and right of the menu box are single rows
   of `▇`, never a full card.

This is visible in the golden file `tui/testdata/TestAppMenuScreenView.golden`:
the rightmost columns show only `▇▇▇▇` stub rows.

### Root cause
The menu background is built by `newFaceDownState` in `tui/menu.go:273-282`,
which deals a normal game and then flips **every** tableau pile's bottom card
face-down:

```go
func newFaceDownState(drawCount int) *engine.GameState {
	state := engine.Deal(engine.NewDeck(), drawCount)
	for _, pile := range state.Tableau {
		if len(pile.Cards) > 0 {
			pile.Cards[len(pile.Cards)-1].FaceUp = false
		}
	}
	return state
}
```

So every tableau column is **entirely face-down** (`FaceUpCards()` returns
`nil`, `FaceDownCount()` equals the full column length).

`RenderTableauPile` in `renderer/pile.go:213-259` only promotes the top
face-down card to a full render when a **drag** is removing all the face-up
cards:

```go
// renderer/pile.go:232
topFDFull := draggingFromHere && len(fuCards) == 0 && fdCount > 0
...
for i := 0; i < fdCount; i++ {
	if topFDFull && i == fdCount-1 {
		rows = append(rows, renderFaceDown(t))   // full 5-row card
	} else {
		rows = append(rows, cardStubTop(t))       // 1-row stub
	}
}
```

Because the menu background passes an empty `renderer.CursorState{}`
(`tui/app.go:273`), `draggingFromHere` is `false`, so `topFDFull` is `false`
and **every** face-down card — including the top one — renders as a 1-row stub.

### Suggested fix
Decouple "promote the top face-down card to a full render" from the drag
condition. The intent is: *whenever a column has no face-up cards, its top
face-down card is the accessible card and should render in full.* In
`renderer/pile.go:232`, change:

```go
topFDFull := draggingFromHere && len(fuCards) == 0 && fdCount > 0
```

to:

```go
topFDFull := len(fuCards) == 0 && fdCount > 0
```

This is safe for live gameplay: after any real move the engine flips the newly
exposed tableau card face-up (see `FlipTableauCardCmd` usage in
`tui/board.go:buildMoveCmd`), so an all-face-down column never occurs during a
game except transiently mid-drag — which this branch already handled. The only
behavioural change is the menu background (and any future all-face-down state)
now shows a full card top.

### Verification
1. `go run .` — the tableau columns behind the menu now show a full face-down
   card body at the bottom of each column, stubs above.
2. `make golden-update` then review `git diff tui/testdata/ renderer/testdata/`.
   Expect `TestAppMenuScreenView.golden` (and any tableau snapshot that includes
   an all-face-down column) to gain full-card rows. Confirm normal-gameplay
   board goldens (`TestRendererFullBoard.golden`, etc.) are **unchanged** —
   those columns always have a face-up bottom card.

## Bug 2 — Foundation piles cannot be reached with the arrow keys

### Symptom
Pressing the arrow keys never lands the cursor on any of the four foundation
piles. They are reachable only via `Tab` / `Shift+Tab`. The expectation is that
arrow-key navigation can move onto the foundations like any other pile.

### How to reproduce
1. `go run .`, start a game.
2. Use only `←`/`→`/`↑`/`↓`. There is no key sequence that places the cursor on
   a foundation. (`Tab` does reach them, confirming the foundations are
   otherwise navigable.)

### Root cause
The Left/Right navigation cycle in `tui/cursor.go:10-20` deliberately omits the
foundations:

```go
// navCycleOrder is the Left/Right arrow key navigation cycle.
// Foundations are omitted — they are reachable only via Tab.
var navCycleOrder = []engine.PileID{
	engine.PileStock,
	engine.PileWaste,
	engine.PileTableau0,
	...
	engine.PileTableau6,
}
```

`MoveLeft` / `MoveRight` (`tui/cursor.go:74-93`) not only skip foundations, they
actively bounce *off* them — if the cursor is somehow on a foundation (e.g.
arrived via `Tab`), `MoveLeft` jumps to `PileWaste` and `MoveRight` jumps to
`PileTableau0`:

```go
func (c *Cursor) MoveLeft(state *engine.GameState) {
	if isFoundationPile(c.Pile) {
		c.Pile = engine.PileWaste
	} else {
		c.Pile = prevInCycle(navCycleOrder, c.Pile)
	}
	...
}
```

`MoveUp` / `MoveDown` (`tui/cursor.go:109-136`) are no-ops unless the cursor is
already on a tableau pile, so they can't move between the top row (stock / waste
/ foundations) and the tableau row either. Net result: foundations are an
arrow-key dead zone. (`tabCycleOrder` at `tui/cursor.go:23-37` *does* include
them, which is why `Tab` works.)

### Suggested fix
Pick the navigation model that matches the 2-D board layout:

```
Row 0:  Stock  Waste            F0 F1 F2 F3
Row 1:  T0  T1  T2  T3  T4  T5  T6
```

Two reasonable options (a future agent should confirm the desired model — see
`docs/DESIGN.md` §8.2/§8.4 which only specifies "left/right between piles"):

- **Option A (simplest):** add the four foundations to `navCycleOrder` after
  `PileWaste`, so Left/Right cycles `Stock → Waste → F0 → F1 → F2 → F3 → T0 …
  → T6 → (wrap)`. Then delete the special-case foundation bounce in `MoveLeft`/
  `MoveRight` (lines 75-79 and 87-91) so foundations behave like any other pile
  in the cycle. This makes `navCycleOrder` identical to `tabCycleOrder` — at
  which point they can be merged.

- **Option B (true 2-D):** keep Left/Right within a row and make `MoveUp`/
  `MoveDown` jump between the top row and the tableau row (e.g. Up from a
  tableau column lands on the nearest top-row pile by x-position; Down returns
  to a tableau column). This is closer to spatial intuition but requires a
  column→pile mapping.

Option A is the smaller change and directly resolves the reported bug.

### Verification
1. Unit test in `tui/cursor_test.go`: assert that repeated `MoveRight` from
   `PileWaste` visits `PileFoundation0..3` (Option A), and that `MoveLeft` from
   a foundation steps to the adjacent foundation/waste rather than bouncing.
2. `go run .` — arrow keys can now land on and leave each foundation pile, and
   the cursor arrow (`↑`) renders beneath the focused foundation.
3. `make check`.

## Bug 3 — Keyboard `Enter` selection teleports the card to (0,0) and shows no indicators

### Symptom
Pressing `Enter` to pick up a card:
- the selected card visually jumps to the **top-left corner (0,0)** of the
  board (a stray card appears there), and
- **no** destination/source highlight is drawn, so there is no feedback about
  what is selected or where it will go.

Expected behaviour (matches `docs/DESIGN.md` §8.4 "Cursor drag-style"):
- On first `Enter`, the card **stays in place** and its pile shows the hover
  arrow (a "selected" indicator).
- Moving the cursor with the arrow keys shows a **second** arrow at the
  prospective destination pile.
- A second `Enter` executes the move and clears all arrows (Esc cancels).

### How to reproduce
1. `go run .`, start a game.
2. Move the cursor to any face-up card and press `Enter`.
3. Observe: a card render appears at the very top-left of the board; the source
   card disappears from its column; no arrows are shown anywhere.

### Root cause
The keyboard "select" flow **reuses the mouse drag state** (`Dragging`,
`MouseX`, `MouseY`) but the renderer interprets that state as a *mouse* drag.

1. **Ghost drawn at (0,0).** `handleSelect` (`tui/board.go:309-334`) sets
   `Dragging = true`, `DragSource`, and `DragCardCount`, but never sets
   `MouseX`/`MouseY` — those are only ever written by the mouse handlers
   (`tui/board.go:154-155, 165-166`) and reset to 0 on drop
   (`tui/board.go:188-189`). The renderer then unconditionally overlays the drag
   ghost at the mouse coordinates whenever `Dragging` is true
   (`renderer/renderer.go:68-71`):

   ```go
   if cursor.Dragging {
       ghost := r.renderGhostCard(state, cursor)
       board = applyOverlay(board, ghost, cursor.MouseY, cursor.MouseX, r.width)
   }
   ```

   With `MouseX == MouseY == 0`, the ghost lands at the top-left corner. This is
   the "jumps to 0,0" symptom.

2. **Source card disappears.** The pile renderers deplete the drag source while
   `Dragging` is true, regardless of input source:
   `RenderTableauPile` slices off the lifted cards
   (`renderer/pile.go:217-223`), `RenderWastePile` hides the top card
   (`renderer/pile.go:108-118`), `RenderFoundationPile` decrements the count
   (`renderer/pile.go:151-159`). For a mouse drag this is correct (the card
   follows the cursor), but for a keyboard selection the card should stay put.

3. **No indicator arrows.** `pileArrowColor` (`renderer/pile.go:67-78`)
   suppresses the cursor arrow while dragging:

   ```go
   if cursor.Pile == pid && !cursor.Dragging {
       return t.CursorBorder, true
   }
   ```

   And the `cardSelected` visual state has no effect on a rendered card —
   `renderCard`/`renderFaceUp` (`renderer/card.go:55-147`) ignore the state
   entirely (only arrows below the pile convey focus). So during a keyboard
   selection: arrows are suppressed *and* there is no on-card highlight, leaving
   zero feedback.

### Suggested fix
Distinguish a **keyboard selection** from a **mouse drag** rather than
overloading one `Dragging` flag. A clean approach:

- Add a `Selecting bool` (keyboard pick-up) field to `Cursor`
  (`tui/cursor.go:41-52`) and `renderer.CursorState` (`renderer/card.go:14-25`),
  set it in `handleSelect` (`tui/board.go:309-334`) instead of / alongside
  `Dragging`, and keep `Dragging` exclusively for the mouse ghost flow.
- In `renderer/renderer.go:68`, gate the ghost overlay on the mouse drag only
  (`if cursor.Dragging` stays, but keyboard selection no longer sets
  `Dragging`), so no ghost is drawn at (0,0) for keyboard selection.
- In the pile renderers, only deplete the source pile for a mouse drag, not a
  keyboard selection, so the picked-up card stays visually in place.
- In `pileArrowColor` (`renderer/pile.go:67-78`):
  - Show the **source** arrow on `cursor.DragSource` while a keyboard selection
    is active (the "card stays put with hover arrow" requirement).
  - Show the **destination** arrow on `cursor.Pile` as it moves (the "second
    hover arrow" requirement), in a distinct colour if desired
    (e.g. `t.CursorBorder` for source, `t.HintBorder`/another for destination).
- Ensure the second `Enter` / `Esc` paths in `handleSelect` and the
  `ActionCancel` handler (`tui/board.go:195-199`) clear the new `Selecting`
  state and all arrows.

Confirm the exact colour/indicator design against `docs/DESIGN.md` §8.4 before
implementing; the spec says held cards "render with a visual indicator
(highlight, different border)" but doesn't pin the arrow scheme.

### Verification
1. `go run .` — press `Enter` on a card: it stays in place, an arrow marks the
   source. Arrow-key to another pile: a second arrow marks the destination.
   Press `Enter`: the move executes (or is silently rejected) and all arrows
   clear. Press `Esc` mid-selection: selection clears, no move. Confirm mouse
   drag-and-drop still works (ghost follows the mouse, source depletes).
2. Snapshot tests in `tui/board_test.go` for: post-`Enter` selection render
   (source arrow, card present, no (0,0) ghost) and cursor-moved-while-selected
   render (source + destination arrows).
3. `make golden-update`, review the diff, then `make check`.
